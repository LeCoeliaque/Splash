# Splashin Switch — Render.com Deployment Guide

## Files
```
splashin-switch/
├── app.py                  # Flask backend (multi-user, session-isolated)
├── requirements.txt        # Python dependencies
├── render.yaml             # Render.com blueprint (auto-configures everything)
└── templates/
    └── index.html          # Frontend UI
```

## Deploy in 4 Steps

### 1. Push to GitHub
Create a new repo (can be private) and push all four files:
```bash
git init
git add .
git commit -m "initial"
git remote add origin https://github.com/YOU/splashin-switch.git
git push -u origin main
```

### 2. Create Render Service
- Go to https://render.com → **New → Web Service**
- Connect your GitHub repo
- Render will auto-detect `render.yaml` and fill in the settings

### 3. Set the ONE Secret Env Var
In the Render dashboard → your service → **Environment**:
```
SUPABASE_ANON_KEY = <your Supabase anon/public key>
```
Everything else is already set in `render.yaml`.
`SECRET_KEY` is auto-generated by Render — you don't need to set it.

### 4. Deploy
Click **Deploy**. First build takes ~2 minutes. You'll get a URL like:
`https://splashin-switch.onrender.com`

## How Multi-User Works

- Each user logs in with their **Supabase email + password only** — no tokens to copy
- The server fetches their location, access token, and refresh token automatically on login
- Every user gets **completely isolated state** (position, route, battery, logs, waypoints)
- A background thread per user handles movement + token refresh + device heartbeat
- Sessions persist via server-side cookies (survive page reloads)
- Inactive sessions auto-clean after 2 hours

## Important Notes

### Worker Configuration
The `render.yaml` uses `--workers 1 --threads 8`. This is intentional:
- **1 worker** = all users share the same in-memory `_user_states` dict
- **8 threads** = handles concurrent requests efficiently
- If you scale to multiple workers/instances, user state would split across processes — you'd need Redis. For typical use (< 50 users), single worker is fine.

### Render Free Tier
Free tier spins down after 15 min of inactivity. The first request after sleep
takes ~30 seconds to wake up. Upgrade to Starter ($7/mo) to keep it always-on.

### Persistent Waypoints
Saved waypoints live in memory and reset on redeploy. To persist them, you can:
- Add a Supabase table `user_waypoints` and load/save on login/change
- Or use Render's persistent disk (add to render.yaml: `disk: { name: data, mountPath: /data, sizeGB: 1 }`) and write a small JSON file per user

export SUPABASE_ANON_KEY="your_key_here"
